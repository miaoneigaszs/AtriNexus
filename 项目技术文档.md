## 1. 项目概述

### 1.1 项目简介

**AtriNexus** 是一个基于企业微信生态的智能对话机器人系统，深度整合了大型语言模型(LLM)、多模态能力、三层记忆系统和高级检索增强生成(RAG)技术。该项目旨在为企业提供一个高度可定制、功能强大且易于扩展的智能助手。

### 1.2 核心特性

#### ✅ 已实现功能

- **多用户支持**: 支持企业微信多用户并发交互
- **沉浸式角色扮演**: 支持单聊场景，可自定义人设
- **图片识别**: 支持Vision Language模型识别用户发送的图片内容
- **持久记忆存储**: 三层记忆架构(短期/中期/核心)
- **知识库管理**: 支持PDF/Word/TXT/MD格式文档
- **智能检索**: 两阶段RAG检索(向量召回+Rerank重排)
- **意图识别**: 基于LLM的智能意图路由
- **网络搜索**: LLM工具调用方式，集成Tavily搜索API
- **可观测性**: Prometheus指标+健康检查端点
- **日记生成**: AI自动生成对话日记

#### 🚧 开发中功能

- OneBot协议兼容
- 1.5版本完全重构

### 1.3 应用场景

- 企业智能客服
- 知识库问答系统
- 个人AI助手
- 角色扮演对话
- 文档智能检索

│  - 时间衰减: 考虑文档新鲜度               │
└────────┬────────────────────────────────┘
         │
         ▼
    注入LLM Prompt

## 3. 核心功能详解

### 3.1 消息处理流程

```python
# 核心流程伪代码
async def process_message(user_id, content, msg_id):
    # 1. 消息去重
    if is_duplicate(msg_id):
        return
    
    # 2. 检查是否为知识库管理命令
    if content in ['知识库列表', '知识库结构', '删除知识库']:
        handle_kb_command()
        return
    
    # 3. 检查是否在KB检索会话中
    if in_kb_search_session(user_id):
        handle_kb_session_response()
        return
    
    # 4. 准备搜索上下文
    ctx = prepare_search_context(user_id, content)
    
    # 5. 执行RAG检索
    kb_results = execute_rag_retrieval(user_id, content)
    
    # 6. 构建完整上下文
    # - 核心记忆
    # - 中期记忆(语义相关)
    # - 短期记忆(最近对话)
    # - 知识库上下文
    
    # 7. 调用LLM生成回复
    reply = llm_service.get_response(...)
    
    # 8. 保存对话记录
    save_to_database()
    
    # 9. 更新记忆
    memory_manager.after_reply(...)
    
    # 10. 发送回复
    send_message(user_id, reply)
```

### 3.2 意图识别机制

**目的**: 智能判断用户消息是否需要查询知识库，避免无效检索。

**实现方式**:
- 使用轻量推理模型（默认 Qwen/Qwen2.5-7B-Instruct）
- 输出结构化JSON，包含意图、置信度、建议分类
- 支持多轮澄清对话

**判断逻辑**:
```
TYPE_KNOWLEDGE_BASE (需要查知识库):
- 用户在寻求具体的、客观的信息或知识
- 问题涉及需要查阅文档才能回答的内容
- 明确的疑问句，期望得到事实性答案
- 例如："报销流程是什么？"、"公司的考勤规定？"

TYPE_CHITCHAT (普通对话):
- 用户在表达个人情感、态度或偏好
- 用户在发号施令或发出控制指令
- 日常问候、闲聊、情感交流
- 实时新闻、天气等需要网络搜索的内容（由LLM工具调用处理）
```

### 3.3 知识库管理

#### 支持的文档格式

| 格式 | 解析方式 | 目录结构识别 |
|------|---------|-------------|
| `.pdf` | PyMuPDF | ✅ 书签目录 + 字体大小识别 |
| `.docx` / `.doc` | python-docx | ✅ 标题样式（Heading 1-6） |
| `.md` | 直接读取 | ✅ Markdown 标题（`#`） |
| `.txt` | 直接读取 | ❌ 无结构 |
| `.csv` / `.json` | 纯文本读取 | ❌ 无结构 |

#### 文档处理流程

```
上传文档
    │
    ▼
┌─────────────────────────────────────────┐
│  文本提取 & 结构识别                      │
│  ├─ PDF: 书签目录 → Markdown标题         │
│  │        无书签时通过字体大小识别标题     │
│  ├─ DOCX: 标题样式 → Markdown标题        │
│  └─ MD: 直接读取，保留 # 标题结构         │
└────────┬────────────────────────────────┘
         │
         ▼
    MD5去重校验
         │
         ▼
┌─────────────────────────────────────────┐
│  结构化拆分 (split_markdown)             │
│  - 按 Markdown 标题层级 (H1/H2/H3) 拆分  │
│  - 保留标题元数据到 metadata             │
│  - 每块携带所属章节信息                  │
└────────┬────────────────────────────────┘
         │
         ▼
    递归切块 (带重叠)
         │
         ▼
    向量化入库 (ChromaDB)
```

### 3.4 图片识别功能

**流程**:
1. 用户发送图片消息
2. 通过企业微信API下载图片(media_id)
3. 图片压缩(>500KB自动压缩至1024x1024)
4. 调用Vision Language模型识别内容
5. 生成图片描述(150字以内)
6. 将描述作为文本消息继续对话流程

**支持的VL模型**:
- DeepSeek-VL
- Qwen-VL
- 其他OpenAI兼容的VL模型

### 3.5 Token监测系统

**功能**:
- 实时记录每次LLM调用的token使用量
- 按用户、模型、请求类型统计
- 估算费用(美元)
- 详细 token 分解（人设、记忆、知识库、历史对话）
- 提供API查询接口

**日志输出格式**:
```
[Token] chat | deepseek-chat | 输入=3615, 输出=44, 合计=3659 | 费用≈$0.001024
[Token详情] 人设=2146 | 记忆=472 | 历史=929 | 用户消息=23
```

**监控指标**:
```python
# Prometheus指标
token_by_type_total      # 按类型统计(prompt/completion)
token_by_user_total      # 按用户统计
token_cost_estimate_usd  # 费用估算
active_llm_requests      # 当前活跃请求数
```

**API鉴权**:
- 可选的 API Key 鉴权中间件
- 通过环境变量 `API_KEY` 启用
- 支持 Header (`X-API-Key`) 或 Query (`api_key`) 传递
- 跳过公开路径：企微回调、健康检查、指标端点、前端页面

### 3.6 网络搜索功能

**集成**: Tavily Search API

**配置**: 在 `config.json` 的 `network_search_settings` 中配置 `api_key` 和 `search_enabled`

**触发方式**: LLM 工具调用（由模型自主决定是否需要搜索网络）

**流程**:
1. 用户询问实时信息（新闻、天气、价格等）
2. LLM 自主判断需要调用 `web_search` 工具
3. 调用 Tavily API 执行搜索
4. 搜索结果返回给 LLM 继续生成回复

**工具定义**:
```python
{
    "name": "web_search",
    "description": "在互联网上搜索实时信息、新闻、天气、当前事件等",
    "parameters": {
        "query": "搜索查询关键词"
    }
}
```

## 6. 核心模块详解

### 6.1 MessageHandler (消息处理器)

**位置**: `src/wecom/handlers.py`

**职责**:
- 接收并处理企业微信消息
- 协调各个服务模块(LLM、RAG、Memory)
- 管理对话流程和上下文
- 处理图片、文本等不同消息类型

**核心方法**:
```python
class MessageHandler:
    async def process_message(user_id, content, msg_id)
        """异步处理消息的核心方法"""
        
    async def process_image_message(user_id, media_id, msg_id)
        """处理图片消息"""
        
    def _execute_rag_retrieval(user_id, content, ...)
        """执行RAG检索"""
        
    def _prepare_search_context(user_id, content)
        """准备搜索上下文(加载记忆)"""
```

### 6.2 LLMService (大语言模型服务)

**位置**: `src/services/ai/llm_service.py`

**职责**:
- 管理与LLM API的交互
- 构建对话上下文和Prompt
- 处理工具调用（时间查询、网络搜索）
- 响应内容清理和过滤
- Token使用统计

**核心特性**:
- **自动模型切换**: 支持配置多个备用模型，失败时自动切换
- **工具调用**: 集成 `get_current_time` 和 `web_search` 工具，模型可自主决定调用
- **思考内容过滤**: 支持DeepSeek-R1等推理模型的特殊输出格式
- **模型兼容性**: 自动跳过不支持工具调用的模型（如 deepseek-reasoner）

**可用工具**:
| 工具名称 | 用途 | 触发场景 |
|---------|------|---------|
| `get_current_time` | 获取当前时间、日期、农历 | 用户询问时间、日期 |
| `web_search` | 网络搜索实时信息 | 用户询问新闻、天气、价格等实时信息 |

**Prompt构建流程**:
```python
final_prompt = base_content         # 基础行为指南
    + worldview_content             # 世界观(可选)
    + core_memory                   # 核心记忆
    + system_prompt                 # 人设提示词
    + kb_context                    # 知识库上下文(可选)

messages = [
    {"role": "system", "content": final_prompt},
    *chat_history,                  # 历史对话
    {"role": "user", "content": message}
]
```

### 6.3 MemoryManager (记忆管理器)

**位置**: `src/services/memory_manager.py`

**职责**:
- 管理三层记忆的存储和检索
- 定期更新核心记忆和中期记忆
- 构建完整的对话上下文
- 支持异步操作

**记忆更新机制**:
```
每10轮对话:
1. 生成对话摘要 → 写入向量库(中期记忆)
2. 更新核心记忆:
   - 使用LLM分析最近对话
   - 提取关键信息(用户信息、偏好、约定)
   - 保留原始核心记忆
   - 添加新的核心记忆
   - 字数控制: 50-100字
```

**时间衰减算法**:
```python
# 中期记忆检索时的时间衰减
decay = max(min_weight, decay_rate ** (hours_diff / 24.0))
final_score = similarity_score * decay

# 默认参数（可在 config.json 的 behavior.memory_decay 中配置）:
decay_rate = 0.98      # 日衰减率 (daily_decay_rate)
min_weight = 0.5       # 最低权重 (min_weight)
```

### 6.4 RAGEngine (检索增强生成引擎)

**位置**: `src/services/rag_engine.py`

**职责**:
- 文档解析和切块
- 向量化入库
- 知识检索(两阶段)
- 文档结构大纲管理

**文档解析策略**:

| 格式 | 目录识别方法 | 说明 |
|------|-------------|------|
| PDF | `doc.get_toc()` | 获取书签目录，转换为 Markdown 标题 |
| PDF | 字体大小分析 | 无书签时，通过 `get_text("dict")` 分析字体大小识别标题 |
| DOCX | 样式检测 | 检测 `Heading 1-6` / `标题 1-6` 样式，转换为 Markdown |
| MD | 正则匹配 | 匹配 `^(#{1,6})\s+(.*)` 识别标题层级 |

**文档切块策略**:
```python
# 递归分层切块
separators = ["\n\n", "\n", "。", "！", "？", ".", " "]

# Markdown结构化拆分 (所有格式统一处理)
- 按标题层级(H1/H2/H3)拆分
- 保留标题元数据到 metadata (H1, H2 等字段)
- 支持按标题过滤检索

# 元数据示例
{
    "file_name": "技术手册.pdf",
    "category": "技术文档",
    "H1": "第一章 系统概述",      # 来自 PDF 书签 / DOCX 标题样式
    "H2": "1.1 系统架构",
    "chunk_index": 5,
    "doc_hash": "abc123..."
}

# 参数配置（可在 config.json 的 behavior 中配置）
chunk_size = 800       # 块大小(字符) - 配置项: rag_chunk_size
chunk_overlap = 150    # 重叠字符数 - 配置项: rag_chunk_overlap
batch_size = 8         # 批量写入大小(智能调整) - 配置项: rag_batch_size
```

## 7. 数据流程

### 7.1 完整对话流程

```
用户发送消息 → 企业微信平台 → FastAPI服务
    ↓
MessageHandler处理
    ↓
消息去重检查 → 加载会话状态 → 意图识别(可选)
    ↓
RAG检索(如果需要) → 加载三层记忆 → 构建完整上下文
    ↓
调用LLM API → 工具调用(可选) → 生成回复 → 保存对话记录
    ↓
更新记忆 → 发送回复 → 用户收到消息
```

### 7.2 记忆更新流程

```
对话结束 → 更新短期记忆(追加到列表)
    ↓
检查对话计数(是否达到10轮?)
    ↓
是 → 生成对话摘要 → 写入向量库(中期记忆)
    ↓
更新核心记忆(LLM分析+摘要) → 重置计数器
    ↓
否 → 计数+1 → 结束
```

### 7.3 RAG检索流程

```
用户查询: "公司的报销流程是什么?"
    ↓
意图识别 → TYPE_KNOWLEDGE_BASE (置信度0.92)
    ↓
查询重写 → "公司报销流程规定"
    ↓
第一阶段: 向量召回(15个候选文档块)
    ↓
第二阶段: Rerank重排(返回Top-3)
    ↓
构建知识上下文 → 注入LLM Prompt
```

## 11. 常见问题

- 降低分类过滤条件

- 对话轮次需达到10轮
- 检查LLM服务是否正常

# Agent 工具编排设计

本文档说明当前 AtriNexus 的 Agent 工具编排方式，以及为什么它与 OpenClaw 第 14 章的思路相似、但保持更轻的实现。

## 1. 设计目标

当前项目的目标不是做通用 AI IDE，而是做一个基于企业微信的智能助手。因此工具编排要满足四个要求：

1. 工具对模型清晰可见，减少误用。
2. 工具集合要随当前消息缩放，避免把所有能力一次性暴露给模型。
3. 写文件和高风险命令要走更保守的路径。
4. 代码保持直接、低嵌套，不引入额外的大型注册中心。

## 2. 当前编排结构

当前链路由三层组成：

1. `WorkspaceRuntime`
   - 负责真正执行目录读取、文件读取、命令执行、预览写入等动作。
   - 是运行时边界，不关心 LangChain。

2. `ToolCatalog`
   - 负责根据用户消息生成工具集合。
   - 负责给模型提供“工具摘要”和“工具实例”。
   - 不做复杂业务决策，只做轻量启发式选择。

3. `LangChainAgentService`
   - 负责构建 agent。
   - 把工具摘要注入 system prompt。
   - 控制最大步数与模型客户端复用。

这三层分别解决“执行”“编排”“推理”，职责比较稳定。

## 3. 参考 OpenClaw 第 14 章后的取舍

OpenClaw 第 14 章最值得借鉴的两点是：

1. 给模型一份“工具摘要”，不要只扔完整 schema。
2. 工具描述要强调“什么时候用”，而不是只写“这是什么”。

当前实现吸收了这两点，但没有照搬 OpenClaw 的完整平台式结构。

我们保留了：

1. `ToolBundle`
   - 同时返回 `tools` 与 `summary_lines`
   - 模型能看到更短的工具导航

2. 面向选择的工具描述
   - 例如 `read_file`、`search_files`、`run_command` 的摘要都直接写明适用场景

3. 动态工具暴露
   - 用户问目录、文件、代码时，才给文件工具
   - 用户表达修改意图时，才给预览写入工具
   - 用户表达命令意图时，才给命令工具

我们没有引入：

1. 大型工具注册中心
2. 工具 schema 二次生成系统
3. 多阶段 planner / executor / reviewer
4. 插件化工具市场

原因很简单：项目体量还不需要这些层。

## 4. 当前工具选择策略

`ToolCatalog.build_tool_bundle(...)` 会读取当前用户消息，按启发式规则决定工具集合。

当前规则分四类：

1. 文件读取意图
   - 匹配文件后缀、目录、代码、配置、日志等关键词
   - 暴露：
     - `list_directory`
     - `read_file`
     - `search_files`

2. 文件修改意图
   - 匹配“修改、替换、追加、创建、覆盖、更新”等词
   - 暴露：
     - `preview_write_file`
     - `preview_edit_file`

3. 命令执行意图
   - 匹配“执行、运行、命令、shell、git、python、pytest、npm、uv、pip”等词
   - 暴露：
     - `run_command`

4. 联网检索意图
   - 匹配“最新、实时、今天、新闻、联网、搜索一下”等词
   - 暴露：
     - `web_search`

始终暴露的工具只有：

1. `get_current_time`

这样做的作用是：

1. 降低模型误调用工具的概率。
2. 降低 prompt 噪声。
3. 让工具编排更接近“当前任务上下文”，而不是全量工具堆叠。

## 5. 为什么使用 `@tool`

当前项目已经使用 LangChain 1.x 的 `create_agent(...)`。在这种前提下：

1. `@tool` 足够表达当前多数工具。
2. 相比一排 `StructuredTool.from_function(...)`，代码更短、更直。
3. 大多数工具只是“把 runtime 的一个能力暴露给模型”，不需要额外包装层。

因此当前策略是：

1. 默认用 `@tool`
2. 只有真的需要复杂 `args_schema` 或动态工具对象时，才考虑回退到 `StructuredTool`

## 6. 为什么仍然保留 `ToolCatalog`

虽然工具本身已经改成 `@tool`，但 `ToolCatalog` 仍然保留，是因为它还承担两件有价值的事：

1. 绑定 runtime
2. 输出工具摘要

如果去掉 `ToolCatalog`，这些逻辑就会散到 `LangChainAgentService` 里，反而让主入口变重。

所以这里保留的是一个很薄的目录层，而不是额外抽象。

## 7. 预览写入而不是直接写入

当前清单选择的是 P1-11A，也就是：

1. 文件修改先生成预览
2. 用户确认后再真正落盘

这样做的原因不是要做 IDE，而是为了降低微信聊天环境中的误改风险。

当前路径是：

1. 模型调用 `preview_write_file` 或 `preview_edit_file`
2. runtime 生成 pending change
3. 用户再通过确认指令触发真正应用

这条路径比直接 `write_file` 多一步，但对聊天式助手更稳。

## 8. 高风险命令策略

命令执行没有做成完整沙箱，但加入了最小风险控制：

1. 普通命令直接执行
2. 明显破坏性命令直接拒绝
3. 高风险但可控命令进入确认流程

同时 pending command 现在绑定 `owner_user_id`，避免跨用户确认。

## 9. Agent 侧约束

`LangChainAgentService` 目前还做了两件关键小约束：

1. 复用 `ChatOpenAI` 客户端
   - 避免每次请求重复创建模型客户端

2. 设置最大递归步数
   - 使用 `ATRINEXUS_AGENT_MAX_ITERATIONS`
   - 避免工具调用循环失控

这两点与工具编排配合后，能让主路径更稳定。

## 10. 当前方案的边界

当前设计是轻量、够用的，但不是平台级方案。它不解决：

1. 通用插件系统
2. 多 agent 协作
3. 工具能力市场
4. 复杂 planner / reviewer / critic 链路

这些不属于当前项目阶段要解决的问题。

## 11. 后续可继续优化的方向

在不破坏当前简洁结构的前提下，后续可继续做：

1. 把工具选择启发式常量进一步收拢
2. 为工具调用失败补更统一的业务日志
3. 给工具编排补少量针对性测试
4. 如果确认预览路径长期保留，再给 pending change 增加更清晰的状态说明

## 12. 一句话总结

当前 Agent 工具编排的核心思路是：

1. runtime 负责执行
2. catalog 负责按当前消息收缩工具集合并生成摘要
3. LangChain 负责推理与调用

它借鉴了 OpenClaw 第 14 章“工具摘要 + 清晰使用说明”的优点，但刻意保持更薄、更直接，更适合 AtriNexus 当前体量。
